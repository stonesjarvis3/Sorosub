# SoroSubs Community Guidelines

## Welcome to the SoroSubs Community! 🎉

We're building a production-ready smart contract for recurring payments on Stellar, and we want YOU to be part of it. This document outlines how we work together as a community.

## Our Documents

### 1. Code of Conduct ([CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md))
**The foundation of our community**

Establishes the values and expectations for all community members:
- **Pledge**: Commitment to a welcoming, harassment-free community
- **Standards**: Examples of good and bad behavior
- **Enforcement**: How we address violations
- **Values**: Inclusivity, transparency, quality, growth

**Key Points:**
- We welcome all backgrounds and experience levels
- Respect, professionalism, and collaboration are essential
- Report violations: conduct@sorosub.example
- Fair investigation and appropriate consequences

### 2. Contributing Guide ([CONTRIBUTING.md](CONTRIBUTING.md))
**How to contribute effectively**

Comprehensive guide for getting involved:
- **Setup**: Environment setup and prerequisites
- **Contribution Types**: Bug fixes, features, docs, testing, DevOps
- **Workflow**: Step-by-step process from issue to merge
- **Guidelines**: Code style, commit messages, PR process
- **Recognition**: How we acknowledge contributors

**Key Sections:**
- Getting started with development environment
- Types of contributions (5 categories)
- Development workflow (6 steps)
- PR guidelines and templates
- Code style guidelines
- Testing requirements

### 3. Wave Program ([WAVE_PROGRAM_PLAN.md](WAVE_PROGRAM_PLAN.md))
**Structured contribution opportunities**

Defines scoped work for contributor engagement:

**Contribution Categories:**
- **Bug Fixes** (20%) - Fix issues and improve reliability
- **New Features** (35%) - Build roadmap items
- **Documentation** (25%) - Create guides and examples
- **Testing & QA** (15%) - Expand test coverage
- **DevOps** (5%) - Deployment and monitoring

**Skill Levels:**
- **Beginner**: Good first issues, documentation, simple tests
- **Intermediate**: Bug fixes, utilities, examples
- **Advanced**: Core features, security, architecture

**Sprint Structure:**
- 2-week development cycles
- Clear issue labels and scoping
- Monthly milestones for major releases

## How to Get Involved

### Level 1: Just Starting Out? 🌱

1. **Read Documentation**
   - Start with [QUICKSTART.md](QUICKSTART.md)
   - Review [README.md](README.md) for overview
   - Check [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md)

2. **Join the Community**
   - Watch the repository
   - Star if you like the project
   - Enable notifications for updates

3. **Try It Out**
   - Follow the 5-minute setup
   - Deploy to testnet
   - Create a test subscription
   - Explore the code

4. **Ask Questions**
   - GitHub Discussions
   - GitHub Issues
   - Email: hello@sorosub.example

### Level 2: Ready to Contribute? 💪

1. **Find an Issue**
   - Look for `good-first-issue` labels
   - Check `wave-program` issues
   - Read the issue description carefully

2. **Set Up Development**
   - Clone the repository
   - Follow setup in [CONTRIBUTING.md](CONTRIBUTING.md)
   - Build and test locally

3. **Make Your Contribution**
   - Create a feature branch
   - Make your changes
   - Test everything
   - Commit with clear messages

4. **Submit a PR**
   - Fill out PR template
   - Link related issues
   - Request review
   - Address feedback

### Level 3: Becoming a Maintainer? 🔧

1. **Consistent Contributions**
   - Multiple merged PRs
   - Quality code and documentation
   - Helpful code reviews
   - Community engagement

2. **Leadership**
   - Review others' PRs
   - Help with issue triage
   - Guide new contributors
   - Share your expertise

3. **Recognition**
   - Join maintainers team
   - Have decision-making input
   - Represent the project
   - Lead initiatives

## Types of Contributions

### 🐛 Bug Fixes
- Issues with clear reproduction steps
- Regression tests to prevent recurrence
- Clear explanation of the fix
- Performance considerations

**Good First Issues**: Yes
**Difficulty**: Varies by complexity

### ✨ New Features
- Align with project roadmap
- Design discussion before implementation
- Comprehensive tests
- Updated documentation

**Good First Issues**: No (usually)
**Difficulty**: Intermediate to Advanced

- Tutorials and guides

- Use case examples
- Translation support

**Difficulty**: Beginner to Intermediate

**Good First Issues**: Yes
**Difficulty**: Beginner to Intermediate

### 🚀 DevOps & Infrastructure

- Docker/Kubernetes setup

- Performance optimization

**Good First Issues**: No (usually)

## Conflict Resolution

### If There's a Disagreement

1. **Communicate Directly**
   - Assume good intent
   - Explain your perspective
   - Listen to theirs

2. **Seek Common Ground**
   - Find areas of agreement
   - Focus on technical merits
   - Look for compromise

3. **Escalate if Needed**
   - Ask maintainers to mediate
   - Present both perspectives clearly
   - Be willing to follow decision

4. **Move Forward**
   - Accept the decision
   - Continue collaboration
   - Learn from the experience

## Code Review Process

### When Reviewing Code

✅ **Do:**
- Ask clarifying questions
- Suggest improvements
- Acknowledge good work
- Be specific and actionable
- Review promptly

❌ **Don't:**
- Be dismissive or rude
- Make it personal
- Require perfection
- Nitpick minor issues
- Delay unnecessarily

### When Your Code is Reviewed

✅ **Do:**
- Thank reviewers for feedback
- Address comments professionally
- Ask for clarification if needed
- Update based on feedback
- Respond in reasonable time

❌ **Don't:**
- Get defensive
- Dismiss feedback
- Make unnecessary changes
- Disappear without responding
- Re-argue decided points
